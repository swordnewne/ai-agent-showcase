# Architecture Diagram Design Skill

> 设计清晰、准确、可读性强的系统架构图。不是"怎么画Mermaid"，是"怎么设计让人一眼看懂的架构"。

## 核心原则（必须背诵）

### 1. 关系诚实

**禁止**：纵向1:1对齐暗示不存在的父子关系。

```
❌ 错误：系统告警 → 社区学习（纵向对齐暗示因果关系）
✅ 正确：所有处理器平级，共享数据层用总线表示
```

**检查方法**：问自己"A在B的正下方，会让读者误以为A触发B吗？"

### 2. 共享服务用总线

多个模块共用同一基础设施时，不要用多对多箭头。

```
❌ 错误：每个处理器分别画箭头到每个数据服务（意大利面）
✅ 正确：执行层 → 共享数据总线 → 数据服务层
```

### 3. 告警/监控独立

监控和告警不是任何业务的子模块，是**横切关注点**。

```
❌ 错误：把告警检查画在金融/心跳下面
✅ 正确：告警和消息/心跳/定时平级，都是事件源
```

### 4. 复杂度控制（Miller's Law）

**单层不超过7个节点**。超过就分组、抽象、拆图。

### 5. 对比度优先

| 背景 | 文字 | 状态 |
|------|------|------|
| `#f9f` (浅粉) | 默认黑 | ❌ 灾难 |
| `#bbf` (浅蓝) | 默认黑 | ❌ 灾难 |
| `#1f6feb` (深蓝) | `#fff` (白) | ✅ 安全 |
| `#21262d` (深灰) | `#c9d1d9` (浅灰) | ✅ 安全 |

**GitHub Dark Mode默认文字是白色**，所以背景要用深色。

## 快速检查清单

画图完成后逐项确认：

- [ ] 纵向对齐是否暗示了不存在的因果关系？
- [ ] 共享服务是否用总线/共享层表示？
- [ ] 告警/监控是否作为独立横切层？
- [ ] 同一层内的节点是否有明确的共同特征？
- [ ] 单层节点数 ≤ 7？
- [ ] 所有文字在目标阅读环境下可读？
- [ ] 可以用一句话描述整张图的核心信息？

## 布局模式速查

| 模式 | 适用场景 | 示意 |
|------|---------|------|
| **分层** | 经典系统架构 | 事件→路由→执行→数据→存储 |
| **总线** | 共享服务 | 多处理器 → 总线 → 多服务 |
| **星型** | 中心节点+卫星 | 网关 → 多个微服务 |
| **管道** | 数据流处理 | 采集→清洗→分析→存储 |
| **矩阵** | 多对多关系 | 用户×权限、服务×环境 |

详细模式说明：[guides/layout-patterns.md](guides/layout-patterns.md)
